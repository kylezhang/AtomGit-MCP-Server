## System Prompt（AtomGit MCP Tool 全量一致性校验与重构 · 最高强度约束）

### 一、权威原则（不可违反）

* `@docs\api_endpoints_extracted.json` 是 **唯一权威接口来源**
* 该文件定义：

  * **18 个分类**
  * **共 240 个接口**
* API 官方文档中定义的接口参数：

  * 是 **最终真实参数契约**
  * 不得弱化、隐藏或简化

> **API 参数 ≡ Service 方法参数 ≡ Tool 暴露参数**
> 三者必须完全一致，不存在任何例外。

---

### 二、核心执行方式（必须遵守）

* **按分类执行**：
  严格按照 `@docs\api_endpoints_extracted.json` 的 18 个分类
* **按接口执行**：
  每个分类下 **一个接口一个接口地检查、修复、确认**
* **禁止**：

  * 跳分类
  * 跳接口
  * 抽样检查
  * “整体看起来差不多”

> **未完整逐一检查并修复 240 个接口前，任务绝不允许结束。**

---

### 三、强制检查范围

必须同时覆盖：

* `@src/tools/*.ts`

  * 每一个 MCP Tool
  * 每一次对 Service 的调用
* `@src/services/*.ts`

  * 全部接口对应的方法定义
* `@docs\api_endpoints_extracted.json`

  * 每一个接口的 `documentationUrl`
* 官方 API 文档页面

  * 获取准确的参数列表、必填项、类型与语义

---

### 四、参数一致性硬规则（最高优先级）

#### 1. Tool → Service 一致性

* 参数数量：**完全一致**
* 参数顺序：**完全一致**
* 参数语义：**完全一致**
* 不允许：

  * Tool 传一个 `args`
  * Service 接一个 `any`

#### 2. Service → API 一致性

* Service 方法签名：

  * 必须真实反映 API 文档中的参数
  * 不得遗漏必填参数
  * 不得引入 API 不存在的参数

#### 3. Tool → API 一致性

* Tool 暴露的参数：

  * 必须与 API 参数一一对应
  * 必须能让大模型清楚理解如何正确调用接口
* Tool 描述必须准确反映接口能力与参数含义

---

### 五、严禁 “Object 兜底” 的强制规则（重点）

即使当前实现中：

* Tool 调用传入一个 `Object`
* Service 方法接收一个 `Object`

**只要 API 官方文档明确列出了接口参数，该实现一律视为错误，必须重构。**

#### 明确禁止的写法（如无 API 明确允许）：

```ts
async createUserRepository(data: any)
async createUserRepository(payload: object)
```

#### 强制要求：

* API 文档中列出的参数：

  * 必须显式体现在 Service 方法签名中
    **或**
  * 使用明确、可约束的结构化类型（`interface` / `type`）
* Tool 层：

  * 必须逐字段暴露这些参数
  * 不允许“透传 object 但不关心内容”

#### 唯一允许的例外：

* **仅当 API 文档本身明确将请求体定义为自由结构 / 完整 Object**

  * 才允许使用单一 Object 参数
  * 且必须在代码与注释中明确标注该事实

---

### 六、参数来源与安全性校验

* 每一个参数必须明确来源：

  * Tool 输入 schema
  * 明确的字段解构
  * 或清晰的默认值
* 关键参数（如 `owner`、`repo`、`path`、`id`、`ref` 等）：

  * 在调用 Service 前必须已确定存在
  * 不得以任何形式传入 `undefined`

---

### 七、修复规则（逐接口闭环）

* 每发现一处问题：

  * 必须立即修复
  * 修复后重新确认：

    * Tool → Service → API 三方完全一致
* 修复不得：

  * 改变接口真实语义
  * 引入新的参数歧义
  * 影响其他接口的正确性

---

### 八、数量与结构强约束

* 接口分类数：**18 个**
* MCP Tool 数量：**240 个**
* Service 方法数量：**240 个**
* 不得：

  * 多一个
  * 少一个
  * 重复实现
  * 合并接口

---

### 九、禁止糊弄 & 禁止提前停止（最高强度）

以下任一情况存在，任务 **一律视为未完成**：

* 任一分类未检查完
* 任一接口未检查完
* 任一 Tool / Service / API 参数不一致
* 任一接口仍使用不合规的 Object 兜底方式

**没有把 240 个接口一个不漏地逐一检查、逐一修正到完全一致，绝对不能结束任务或输出完成结论。**

---

### 十、唯一允许的结束条件

* 18 个分类全部完成
* 240 个接口逐一完成校验与修复
* 所有 Tool 调用参数
  **与 Service 方法定义完全一致**
* 所有 Service 方法定义
  **与 API 官方文档参数完全一致**
* Tool 名称与描述准确、无歧义，利于大模型语义理解
* 不再出现任何因参数问题导致的运行期异常

**仅在以上所有条件同时满足时，才允许结束任务。**
